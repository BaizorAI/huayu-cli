# PRD: huazhen 2026-07-05B 编码修复与端到端验收

## Problem Statement

huazhen 已经具备本地 Rust CLI/TUI 工作台的主要产品形态：用户可以登录 baizor.com 获取 API Key，通过 TUI 输入自然语言任务，并由 huazhen 调起 Codex 或 Claude Code 执行。当前代码中也已经存在配置加载、登录轮询、模型配置、工具安装、PTY 子进程、slash command、状态栏、设置弹窗和统一输出面板等模块。

但当前版本距离可交付仍有三个集中风险：

- 用户可见文案、状态图标、帮助文本和源码注释中存在明显乱码，Windows PowerShell、Windows Terminal 和跨平台终端下的可信度不足。
- 关键能力虽然已经在模块中出现，但缺少从 CLI/TUI 入口到配置文件、工具进程、状态变化和用户提示的端到端验收闭环。
- 测试覆盖主要不足以保护真实用户路径，后续修复编码、工具调用或交互时容易引入回归。

本 PRD 定义 2026-07-05B 的交付目标：不扩大产品范围，优先把现有能力修到用户可读、行为可验收、测试可保护的状态。

## Solution

2026-07-05B 聚焦一次稳定化交付：

- 修复所有用户可见中文、符号和终端文案乱码，保证源码、CLI 输出、TUI 输出和帮助信息统一使用 UTF-8。
- 梳理 CLI 与 TUI 共享的行为边界，让登录、状态查看、模型切换、工具切换、工具更新和错误提示在两个入口下保持一致。
- 为配置隔离、登录流程、slash command、工具可用性检测、工具事件解析、任务取消、滚动行为和设置保存补齐面向外部行为的测试。
- 建立本地验收命令，确保每次交付前可以用 `cargo test`、`cargo fmt`、`cargo clippy` 和手动 TUI 检查验证主路径。
- 保持当前 Rust 技术栈和模块边界，不引入数据库、Web UI 或新的 AI 工具。

## User Stories

1. As a new user, I want all first-run login text to render correctly, so that I can understand how to finish setup.
2. As a returning user, I want the status bar to show readable tool, model, directory, and connection state, so that I know what context huazhen is using.
3. As a TUI user, I want `/help` to show readable commands and shortcuts, so that I can discover available actions without leaving the app.
4. As a CLI user, I want error messages to render correctly, so that I can recover from failures without guessing.
5. As a Windows user, I want Chinese text, borders, symbols, and status markers to display consistently in PowerShell and Windows Terminal.
6. As a macOS or Linux user, I want the same text and shortcut semantics to work in my terminal.
7. As a user, I want `/login` and `huazhen login` to save the same config, so that CLI and TUI setup never diverge.
8. As a user, I want `/status` and `huazhen status` to report the same API Key presence, base URL, model, active tool, and tool availability.
9. As a user, I want `/model <name>` to persist the selected model, so that the next task and next app launch use my choice.
10. As a user, I want tool switching to cancel any running task cleanly, so that only one child process owns the UI state.
11. As a user, I want missing Codex or Claude binaries to produce a clear recovery path, so that I know whether to run `/update`.
12. As a user, I want tool update progress to appear in the main output panel, so that background install work is visible.
13. As a user, I want auth failures to be separate from network failures, so that I know whether to log in again or retry later.
14. As a user, I want task completion to show elapsed time, so that I can understand how long the tool ran.
15. As a user, I want scroll state to remain stable while reading previous output, so that new log lines do not yank the view unexpectedly.
16. As a user, I want Page Up, Page Down, mouse wheel, Up, Down, Enter, Esc, Tab, and Alt+Q to behave predictably.
17. As a user, I want settings changes to save model and base URL together, so that future commands use the updated values.
18. As a developer, I want config reads and writes to be testable under a temporary config root, so that tests never touch my real home directory.
19. As a developer, I want login polling to be deterministic in tests, so that pending, success, timeout, malformed response, and network failure paths are covered.
20. As a developer, I want slash command parsing tested independently from terminal rendering, so that command behavior is stable.
21. As a developer, I want tool event parsing tested with representative Codex and Claude output, so that status transitions are reliable.
22. As a developer, I want PTY spawn failures to be tested at the boundary, so that missing tool errors remain user-facing.
23. As a developer, I want cancellation behavior tested without relying on real long-running AI calls, so that regressions are cheap to catch.
24. As a maintainer, I want a small release checklist, so that the project can be handed to another agent or human with clear acceptance criteria.

## Implementation Decisions

- Keep the existing module split: CLI command handlers, config service, login service, installer service, tool adapter/process layer, TUI app state, and TUI renderer.
- Treat UTF-8 text correctness as a product requirement. Replace mojibake in user-facing strings, help text, status labels, error messages, and comments that explain behavior.
- Prefer plain readable Unicode only where terminals commonly support it. If a symbol is risky on Windows, provide an ASCII fallback or use simple text.
- Keep all persisted huazhen configuration under the huazhen-owned config root. Codex and Claude config directories remain isolated from user-managed tool config.
- Make CLI and TUI call shared services for config, status, tool availability, and login persistence instead of duplicating business rules.
- Keep the TUI main output panel as the single dynamic stream for logs, AI output, events, errors, update progress, and completion messages.
- Keep the right-side panel static except for lightweight state fields such as active tool, model, working directory, and recent commands.
- Preserve current keyboard semantics: Enter sends or confirms, Esc cancels or closes, Tab switches tools, Alt+Q quits, Page Up/Page Down and mouse wheel scroll output, Up/Down navigate input history.
- Do not add conversation persistence in this slice. Runtime message history and session input history are sufficient for 2026-07-05B.
- Do not add new AI tool providers in this slice. Codex and Claude remain the only supported tool types.
- Error categories must stay explicit: not configured, auth error, network error, missing tool, process spawn failure, generic execution error.
- Debug logs may remain under the huazhen directory, but user-facing output must be concise and readable.

## Testing Decisions

Good tests should assert externally observable behavior: files written, config values loaded, CLI output, command parsing results, app state transitions, emitted user-facing messages, tool event classification, and process/session lifecycle. Tests should avoid asserting private widget internals or exact ratatui pixel layouts unless a rendering regression is the behavior being protected.

- Test default config loading when no config exists.
- Test config save/load round trip under a temporary config root.
- Test Codex and Claude config writers only write under huazhen-managed directories.
- Test API Key masking for empty, short, normal, and unusual key lengths.
- Test `huazhen status` output for configured, unconfigured, missing tool, and locally installed tool states.
- Test `huazhen login` success and failure paths with mocked HTTP or injectable login client behavior.
- Test login URL and token generation format.
- Test login polling for pending, success, timeout, malformed response, and network failure.
- Test slash command parsing for `/help`, `/login`, `/status`, `/model`, `/switch`, `/update`, `/install`, `/clear`, `/quit`, and unknown commands.
- Test that `/model <name>` updates config and settings input state.
- Test that switching tools persists active tool and cancels a running process.
- Test that missing tool spawn maps to a missing-tool status and a readable recovery message.
- Test tool event parsing for file writes, test pass, test fail, auth error, network error, generic output, and completion.
- Test input history navigation boundaries and draft restoration.
- Test scroll behavior for append while auto-scroll is enabled and while reading older output.
- Test update progress channel handling, including completion cleanup.
- Test settings apply behavior for model and base URL.
- Test that all checked-in Rust source files are valid UTF-8 and no known mojibake patterns remain in user-facing strings.

## Acceptance Criteria

- `cargo fmt --check` passes.
- `cargo clippy --all-targets --all-features -- -D warnings` passes, or any remaining warning is explicitly documented with a reason.
- `cargo test` passes.
- Running `huazhen status` prints readable text and never exposes a full API Key.
- Starting the TUI with no API Key shows a readable not-configured or login path.
- `/help`, `/status`, `/model`, `/switch`, `/update`, and `/login` all produce readable output in the TUI.
- A missing Codex or Claude binary tells the user to run `/update` or explains the fallback path.
- Auth and network failures are distinguishable in both internal state and user-facing output.
- Source files no longer contain visible mojibake in user-facing strings.
- Manual smoke testing is completed in Windows Terminal or PowerShell.

## Out of Scope

- Adding a Web UI.
- Adding providers beyond Codex and Claude.
- Persisting full conversation history across restarts.
- Multi-account or profile management.
- Changing the baizor.com API contract.
- Replacing ratatui/crossterm.
- Building a full self-update mechanism for huazhen itself.
- Refactoring unrelated modules that do not affect the 2026-07-05B acceptance path.

## Further Notes

This PRD intentionally treats text correctness as part of functionality. For a terminal product, unreadable output is not cosmetic: it blocks setup, recovery, trust, and supportability.

The preferred test seams are the shared service modules and TUI app-state methods. Renderer-level tests should be added only where the behavior cannot be protected through state or command-level tests.
