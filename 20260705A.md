# PRD: huazhen 2026-07-05A 稳定化验收

## Problem Statement

huazhen 已经具备 Rust CLI/TUI 工作台的核心形态：用户可以登录 baizor.com 获取 API Key，在本地 TUI 中输入需求，并由 huazhen 调用 Codex 或 Claude Code 执行任务。当前 PRD 和计划文档已经覆盖了整体产品方向、配置隔离、PTY 子进程、登录、模型、工具切换和界面布局。

但当前交付风险仍然集中在“功能已经存在，但缺少可验收闭环”：

- CLI 与 TUI 的入口能力需要统一，避免同一能力在不同入口下表现不一致。
- 登录、配置保存、模型选择、工具选择、工具执行等主路径需要明确验收标准。
- 子进程输出、错误分类、取消、退出、滚动和输入历史需要有外部可观察的行为定义。
- Windows 终端中文、边框、快捷键和路径行为需要避免乱码和误触。
- 测试需要覆盖真实用户路径，而不是只验证内部函数细节。

## Solution

本 PRD 将 huazhen 的 2026-07-05A 工作定义为一次稳定化验收：不扩大产品范围，优先把已有 CLI/TUI 能力收敛成可运行、可测试、可发布的本地 AI 编程工作台。

交付结果应满足：

- 用户首次启动时可以完成登录并持久化配置。
- 用户再次启动时可以直接进入工作台并看到当前工具、模型和连接状态。
- 用户可以在 TUI 中输入自然语言需求，并由当前工具执行。
- 用户可以通过 CLI 子命令完成登录、状态查看、模型查看或工具更新等非交互操作。
- 用户可见错误有清晰分类：认证失败、网络失败、工具缺失、工具执行失败。
- 所有配置写入 huazhen 管理目录，不污染用户已有 Codex 或 Claude 配置。
- 关键路径都有测试覆盖，测试通过临时目录和 mock 服务运行。

## User Stories

1. As a new user, I want huazhen to guide me through login on first launch, so that I can start without manually editing config files.
2. As a returning user, I want huazhen to load saved config automatically, so that I can start a task immediately.
3. As a user, I want the status bar to show the active tool, model, working directory, and connection state, so that I know what context my task will run in.
4. As a user, I want to send a natural-language prompt from the TUI, so that I can delegate programming work without learning a new command syntax.
5. As a user, I want task output, tool logs, file events, errors, and completion messages in one main output area, so that I can follow execution without scanning multiple live panels.
6. As a user, I want a static help/reference area, so that I can check shortcuts and current settings without interrupting the task.
7. As a user, I want Enter to send input or confirm the active overlay, so that the primary action is predictable.
8. As a user, I want Esc to cancel a running task or close an overlay, so that I can back out consistently.
9. As a user, I want Alt+Q to quit, so that accidental single-key text input cannot close the app.
10. As a user, I want Up/Down to navigate input history, so that I can reuse previous prompts.
11. As a user, I want Page Up/Page Down and mouse wheel scrolling for the main output, so that history reading does not conflict with prompt editing.
12. As a user, I want `/help` to print the available slash commands, so that I can discover supported actions from inside the TUI.
13. As a user, I want `/model <name>` to update the active model and persist it, so that future tasks use the model I selected.
14. As a user, I want `/switch codex` and `/switch claude` to change the active tool, so that I can choose the best tool for a task.
15. As a user, I want tool switching during a running task to cancel the old task cleanly, so that two tool processes do not compete for the same UI state.
16. As a user, I want authentication failures to clearly suggest logging in again, so that I know the next recovery step.
17. As a user, I want network failures to be distinguished from tool failures, so that I know whether to retry later or inspect the local tool.
18. As a user, I want missing tool errors to suggest installation or update actions, so that I can recover without searching documentation.
19. As a CLI user, I want `huazhen login` to perform the same login flow used by the TUI, so that I can configure huazhen in non-TUI environments.
20. As a CLI user, I want `huazhen status` to show masked API Key presence, base URL, default model, active tool, and tool availability, so that I can inspect setup quickly.
21. As a CLI user, I want model-related commands to use the same config as the TUI, so that CLI and TUI never diverge.
22. As a developer, I want all config file operations redirected by a test config root, so that automated tests never write into a real user home directory.
23. As a developer, I want login polling to be testable with mocked HTTP responses, so that success, pending, timeout, malformed response, and network failure paths are deterministic.
24. As a developer, I want command parsing tested independently from terminal rendering, so that slash commands are stable across terminal backends.
25. As a developer, I want tool event parsing tested with representative output lines, so that status transitions are reliable.
26. As a developer, I want cancellation and completion behavior tested at the process/session boundary, so that the UI state does not get stuck.
27. As a developer, I want Windows-compatible text and fallback symbols, so that PowerShell and Windows Terminal remain readable.

## Implementation Decisions

- Keep the existing Rust stack: clap for CLI commands, ratatui/crossterm for TUI, tokio/reqwest for async HTTP, and portable-pty for tool subprocess execution.
- Treat TUI and CLI as two entry points over shared services, not separate implementations. Login, config, model selection, tool resolution, and status inspection should use the same underlying modules.
- Store huazhen-owned config under the huazhen config root. Tests must be able to override this root with an environment variable or injected path.
- Keep Codex and Claude configuration isolated from user-managed system config. huazhen writes only to its own managed directories and injects environment variables when spawning tools.
- Prefer bundled or managed tool resolution first, then PATH as a fallback for advanced users.
- Preserve a single active tool and model in config. Tool switching updates runtime state and persists the selected tool when appropriate.
- Use a single main output stream in the TUI for task output, tool logs, structured events, errors, and completion summaries.
- Keep the right-side panel static except for lightweight state fields such as active tool, model, working directory, and recent commands.
- Use explicit keyboard mapping: Enter for send/confirm, Esc for cancel/close, Alt+Q for quit, Page Up/Page Down and mouse wheel for output scrolling, Up/Down for input history.
- Parse tool output conservatively. Recognize important categories such as file written, test run, test passed, test failed, auth error, network error, generic error, and done; leave unrecognized output as plain log text.
- Surface errors in user-facing language. Auth, network, missing tool, and execution errors should be distinguishable in both TUI output and CLI status paths.
- Avoid adding persistence for conversation history in this PRD. Runtime input history is enough for the 2026-07-05A stabilization pass.

## Testing Decisions

Good tests should assert externally observable behavior: config files written, CLI output, command parsing results, channel events, status transitions, process/session lifecycle, and user-facing error classification. Tests should avoid asserting ratatui pixel-level rendering or private widget internals.

- Test config default loading when no config exists.
- Test config save/load round trip using an overridden config root.
- Test that Codex and Claude managed config writers only write under the huazhen config root.
- Test API Key masking for short, normal, and missing keys.
- Test login URL generation and token format.
- Test login polling success, pending, malformed response, timeout, and network failure with mocked HTTP.
- Test model list fetching and parse failure with mocked HTTP.
- Test CLI status output for configured, unconfigured, missing tool, and auth failure states.
- Test slash command parsing for `/help`, `/model`, `/switch`, `/status`, `/login`, and unknown commands.
- Test input history navigation boundaries and draft restoration.
- Test output scrolling state transitions for Page Up, Page Down, mouse wheel up, and return-to-bottom behavior.
- Test tool event parsing for Codex and Claude representative output.
- Test task cancellation while a PTY session is running.
- Test tool switching while a task is running.
- Test that Windows-safe UI text paths avoid invalid UTF-8 or unsupported control assumptions.

## Out of Scope

- Adding new AI tools beyond Codex and Claude.
- Adding a web UI.
- Persisting full conversation history across restarts.
- Adding multi-account or multi-base-url profile management.
- Implementing a database.
- Building a full auto-update system for huazhen itself.
- Replacing the current Rust TUI architecture.
- Refactoring unrelated modules that do not affect the stabilization acceptance path.

## Further Notes

This PRD is intentionally acceptance-oriented. Existing PRDs describe the larger product and layout direction; this document defines the smallest coherent stabilization slice that can be handed to an implementation agent and verified with focused tests.
